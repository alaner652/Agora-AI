# Agora 產品分層與可售功能規劃

> 內部產品/商業規劃文件（不對外服務；對外技術文件在 `public/`）。
> 狀態：草案。落地前仍有兩個前置決策待拍板（見文末「待確認」）。

## 背景與定位

- **商業模式約束**：學校只願意「免費採用」，不付費；唯一可能付費的是**學生**。
- **法律前提**：因為學校（即使只是非正式）背書，原本「替學生保管帳密、未授權爬蟲」的最大法律地雷**大幅解除**——這讓向學生收費變得可行且相對安全。
- **收入定位要務實**：單一學校、學生價格敏感、轉換率約 2–5%，直接訂閱收入頂多**打平成本 + 零用錢**。真正的機會在於「**學校免費背書 = 白拿全校學生分發**」，未來複製到多校放大。
- **與學校的紅線**：基本查詢功能必須**免費**，否則違背「免費幫學校做」的默契。付費只能加在**主動 / 預測 / 進階**這類加值功能上。

## 分層總覽

| 層 | 對象 | AI 額度 | 加值功能 | 定位 |
|----|------|---------|----------|------|
| **Free** | 全校學生（學校期待免費） | 共用 LLM，**每日小額** | 全部**查詢** + 基本 AI 對話 | 不能 gate，維持學校關係 |
| **Plus**（付費） | 願付費的學生 | 共用 LLM，**較高上限（仍有上限，非吃到飽）** | 洞察類 + 主動推播 + 進階 AI | 主要收入來源 |
| **BYOK** | 進階/重度使用者 | **自備金鑰，無限** | 同 Plus 的加值功能 | 把推論成本轉嫁回使用者 |

**一句話**：Free 夠用、Plus 更多額度 + 加值、BYOK 才吃到飽。

## Token / 額度策略（核心）

現況（已實作）：
- `FREE_DAILY_PER_USER` / `FREE_DAILY_GLOBAL` 控制共用 LLM 的**每人/全站每日上限**，以「每則新訊息」計（[backend/src/storage/usage.py](../backend/src/storage/usage.py)）。
- 超額或無金鑰 → `/chat` 回 402 引導 BYOK；BYOK 一律放行（[backend/src/api/app.py](../backend/src/api/app.py) `_enforce_llm_quota`）。

要支援分層，需要的演進：

| 項目 | 現況 | Plus 需要 |
|---|---|---|
| 額度依方案不同 | 單一固定上限 | **per-plan 上限**：Free 低、Plus 高（仍有頂） |
| 計量單位 | 每則訊息 | 建議改/增 **token 計量**（使用者講「token 用量」、也更貼近真實成本）。`llm_call` log 已記 `prompt_tokens`/`completion_tokens`，計量可行 |
| 方案歸屬 | 無 | 每個 uid 一個 `plan`（free/plus/byok）+ 到期日 |

**建議的額度設計**：
- **Free**：每日 N 則（或每日 X token），小額試用。
- **Plus**：**每月 token 預算池**（例如 Free 的 5–10 倍），用完即停或降級回 Free 額度——**明確封頂，不是吃到飽**。月計比日計更符合「付月費給一個月用量」的直覺。
- **BYOK**：不計量、直接放行（已實作）。

> 實作影響：新增 `user_plan` 儲存（plan + 到期）、把 `record_and_check` 改成**讀方案決定上限**、（選做）改成 token-based 計量。這是 Plus 上線的最小後端改動。

## 可售功能盤點（依 痛感 × 願付 × 可行性）

### 🟢 第一類：算出來的洞察 —— 不需背景存取，用已抓到的資料即可（**最該先做**）

| 功能 | 為什麼有人付 | 可行性 | 層 |
|---|---|---|---|
| **缺曠紅線預警**「這門課還能缺 X 節就被當」 | 被當 = 重修 = 真金白銀，痛感最高 | 高（缺曠已會查 + 學校 1/3 規則計算） | Plus |
| **GPA 試算 / 期末回推**「期末幾分能保 B」 | 成績焦慮普遍 | 高（成績已會查，純計算） | Plus |
| **歷年成績彙整 / 趨勢 / 排名估算** | 想看全貌、申請用 | 高（多學期聚合） | Plus |
| **匯出**（成績單 PDF、課表 ICS） | 申請、備份、貼行事曆 | 高 | Plus |

### 🟡 第二類：主動推播 —— **需要背景存取**（拍板存帳密後才做，但這才是真正的付費鉤子）

| 功能 | 為什麼有人付 | 依賴 | 層 |
|---|---|---|---|
| **成績一出就通知** | 不用一直手動刷 | 背景輪詢 | Plus |
| **缺曠變動 / 逼近上限警示** | 接住紅線、主動推 | 背景輪詢 | Plus |
| **假單審核狀態變更通知** | 等審核焦慮 | 背景輪詢 | Plus |
| **截止 / 考試提醒 + 行事曆同步** | 忘記 = 慘 | 背景 +（可能需補資料來源） | Plus |

> 推播管道：延伸既有 webhook 告警那套，或接 LINE Notify / Discord / Web Push。

### 🔵 第三類：AI / 額度 —— 結構已具備

| 功能 | 說明 | 層 |
|---|---|---|
| **較高對話額度** | Plus 的月 token 池（**封頂**） | Plus |
| **自備金鑰無限**（BYOK） | 已實作，最高階 | BYOK |
| **進階 AI**：讀課程大綱、規劃讀書計畫、考古題問答 | 需使用者上傳資料（檔案上傳已具備） | Plus |

## 關鍵技術前提：背景存取

🟡 推播類全部需要「使用者不在線上時，後端能定時替他查校務系統」。但現況是 **JSESSIONID 只在記憶體、2 小時清、不存密碼**（`refresh_fn=None`），代表**目前零背景能力**。解鎖選項：

| 做法 | 解鎖 | 代價 |
|---|---|---|
| **opt-in 加密存帳密**（沿用 [user_settings.py](../backend/src/storage/user_settings.py) 的 Fernet 模式），僅對開通推播者、明確告知 | 完整背景推播 | 「為營利保管密碼」，信任/責任跳級 |
| Session-only + 定期請重新登入 | 半套（僅在線時提醒） | 體驗差、推播不可靠 |
| 不碰背景，只做 🟢 洞察 | 預測/統計 | 無 push，黏著度較低 |

**這是 Plus 推播功能的前置決策**，且建議**留到驗證有人付費之後再拍板**，別一開始就扛保管密碼的責任。

## 開發順序（Roadmap）

1. **先做 🟢 第一類**（缺曠紅線 + GPA 試算）——零背景依賴、用現有資料、馬上能當 Plus 賣點，**驗證學生到底付不付**。
2. **接 Plus 額度機制**（`user_plan` + per-plan 上限 + 金流）——把 BYOK 旁邊補上「付費較高額度」這一階。
3. **跟學校確認**向學生收費 OK（見待確認）。
4. 驗證有人付 → **再拍板 opt-in 存帳密**，做 🟡 推播（最貴的信任成本留到最後）。
5. （遠期）多校複製：學校免費背書 + 學生 freemium。

## 待確認（落地前必清）

1. **學校關係性質**：是「官方掛名採用」還是「默許你自部署」？這決定能否名正言順向學生收費。
2. **是否做 opt-in 加密存帳密**：決定 🟡 推播整類做不做得了。建議驗證 🟢 有人付費後再定。
3. **計量單位**：Plus 用「月 token 池」還是「每日訊息數」？建議月 token 池（貼合成本與直覺）。
4. **金流**：學生付費走哪個管道（訂閱？一次性？），需另評估。
